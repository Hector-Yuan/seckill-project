# 秒杀系统示例项目（Go + Gin + MySQL + Redis）

> 一个练习高并发秒杀场景的学习项目，包含基础用户系统、商品查询、下单逻辑，以及 Redis 分布式锁与压测程序。

---

## 一、项目简介

本项目是一个简化版的“秒杀/抢购”系统，主要目标是帮助你理解并实践以下内容：

- 如何用 Go + Gin 搭建 Web API 服务；
- 如何用 MySQL + 事务 + 行级锁防止超卖；
- 如何用 Redis 做缓存与分布式锁，防止重复下单、减轻数据库压力；
- 如何通过自制压测脚本模拟高并发场景，观察系统行为。

项目结构大致如下（只列出核心模块）：

- `main.go`：程序入口，初始化路由、数据库、Redis 等；
- `controller/`：HTTP 控制器（业务入口）：
  - `order.go`：创建订单（秒杀下单逻辑）；
  - `product.go`：商品列表、详情查询（含 Redis 缓存）；
- `middleware/`：
  - `JWTAuthMiddleware.go`：JWT 鉴权中间件，负责从请求头解析用户信息；
- `common/`：
  - `database.go`：MySQL 初始化；
  - `redis.go`：Redis 客户端初始化；
- `model/`：
  - `models.go`：User / Product / Order 等数据模型定义；
- `utils/`：
  - `jwt.go`：JWT 生成与解析工具；
- `distributed-lock-renewal.md`：分布式锁续期机制的说明文档。

## 二、技术栈与依赖

- 语言：Go 1.20+（建议使用最新稳定版本）
- Web 框架：Gin
- 数据库：MySQL
- 缓存与分布式锁：Redis
- 配置管理：可能使用 Viper（具体以 `go.mod` 和代码为准）
- 鉴权：JWT（JSON Web Token）

你可以通过查看 `go.mod` 了解完整依赖列表。

## 三、功能概览

### 1. 用户相关

- 用户注册：创建新用户；
- 用户登录：返回 JWT Token，用于后续接口鉴权；
- JWT 中间件会从 `Authorization: Bearer <token>` 中解析用户信息，注入到 Gin 的 `Context` 中。

### 2. 商品相关

- 查询商品列表 / 详情；
- 对商品列表数据做了 Redis 缓存：
  - 优先从 Redis 读取；
  - 未命中时从 MySQL 查询并回填到 Redis，设置适当 TTL。

### 3. 订单 / 秒杀相关

- 创建订单（秒杀下单）：
  - 需要登录（JWT 鉴权）；
  - 使用 Redis 分布式锁限制“同一用户对同一商品的频繁下单”；
  - 使用 MySQL 事务 + 行级锁防止库存超卖；
  - 针对 user_id + product_id 建立唯一索引，用于防止重复购买。

## 四、快速开始

### 1. 环境准备

- 安装 Go（1.20+）
- 安装并启动：
  - MySQL
  - Redis

项目中如果包含 `docker-compose.yml`，也可以选择直接使用 Docker 启动依赖服务：

```bash
docker-compose up -d
```

请确保：

- `main.go` 中数据库和 Redis 的连接配置与实际环境一致；
- 如果使用 Docker，端口和账号密码与 `docker-compose.yml` 中保持一致。

### 2. 拉取依赖

在项目根目录执行：

```bash
go mod tidy
```

### 3. 启动 Web 服务

在项目根目录执行：

```bash
go run main.go
```

默认情况下，Gin 会在例如 `:8080` 端口监听（以实际代码为准）。  
你可以使用 curl / Postman / 浏览器来访问接口。

### 4. 运行压测程序

在项目根目录执行：

```bash
go run ./cmd/stress_test
```

压测程序会：

- 创建一批测试用户；
- 使用这些用户并发请求登录获取 Token；
- 使用 Token 并发调用下单接口；
- 打印成功 / 失败的统计结果。

根据压测结果，你可以调整：

- 并发用户数；
- 商品初始库存；
- 是否开启 / 关闭 Redis 分布式锁；
- SQL 事务的使用方式；
以便对比不同方案下的行为和稳定性。

## 五、核心设计说明（简要）

### 1. 防止超卖：MySQL 行级锁 + 事务

在订单创建逻辑中（参考 `controller/order.go`）：

- 使用事务（`BEGIN` / `COMMIT` / `ROLLBACK`）；
- 对库存记录使用 `SELECT ... FOR UPDATE` 或 ORM 等价写法；
- 在事务内检查库存、扣减库存、写入订单记录；
- 保证在数据库层面不会出现“库存被多次扣减”的情况。

这一部分是最终一致性的核心保障。

### 2. 防止重复下单 / 频繁操作：Redis 分布式锁

秒杀场景下，常见需求包括：

- 限制同一用户短时间内多次点击“下单”按钮；
- 做简单防刷限流。

项目中使用 Redis 作为分布式锁实现：

- 锁 key 通常形如：`order_lock:{user_id}:{product_id}`；
- 使用 `SETNX` 或 `SET NX PX` 成功获得锁后，才继续订单逻辑；
- 若加锁失败，直接返回“请勿频繁操作 / 重复下单”的提示；
- 在逻辑结束后删除锁，或依赖短 TTL 自动释放。

在更复杂场景下，可以参考 `distributed-lock-renewal.md` 中的“续期机制”设计，让锁在长耗时任务中也保持安全。

### 3. 缓存与回源

商品列表接口使用 Redis 做缓存：

- 客户端请求商品列表；
- 服务端优先从 Redis 获取；
- 若没有数据，则从 MySQL 查询；
- 将结果序列化后写回 Redis，设置 TTL；
- 后续请求会直接走 Redis，加快响应速度并减轻数据库压力。

## 六、可能的扩展方向

如果你想继续在本项目上深入学习，可以尝试：

- 增加“订单状态流转”（待支付、已支付、已取消等）；
- 引入消息队列（Kafka / RabbitMQ）做“异步下单 / 削峰”；
- 在压测工具中引入更详细的统计（QPS、RT、错误类型分布等）；
- 实现完善的日志和链路追踪（如使用 zap、OpenTelemetry）；
- 对分布式锁实现做抽象封装，支持不同后端（Redis / etcd 等）。

## 七、注意事项

- 本项目主要用于学习和演示，并非工业级完整方案；
- 在真实生产环境中，还需要考虑：
  - 安全认证与权限控制；
  - 数据库读写分离与分库分表；
  - Redis 集群高可用；
  - 更严格的限流、风控策略；
  - 灰度发布、监控告警、日志收集等。
